# 调研工作流：从「调研 X」到一篇可决策的文档

调研文档失败的最常见原因不是文笔，而是**没先把问题界定清楚就开始搜，或者搜了一堆二手资料就下笔**。这份工作流的目的就是按顺序卡住这两个坑。

---

## 阶段 0 · 界定（接到请求后最先做）

在动手搜任何东西之前，先确定三件事：

| 要素 | 自问 | 为什么重要 |
|---|---|---|
| **对象** | X 是产品 / 源码仓库 / 技术方案 / 可行性 / 现实事务 / 知识体系（书·理论·思想家·学科——做导读）？ | 决定取证手段（见阶段 1） |
| **读者与用途** | 给谁看？支撑什么决策？ | 决定深度、侧重、术语密度 |
| **关键问题** | 这篇必须回答哪 3–5 个问题？ | 没有问题清单的调研一定发散 |

- 信息不全时，**做合理假设并写进文档「范围 / Scope」一节**，不要停下来反复追问用户。
- 关键问题清单就是后面正文「核心分析」的分节依据。

**反例**：用户说「调研一下抖音收藏 API」。错误做法是直接搜「抖音 API 文档」开始抄。正确做法是先锁定关键问题——「官方到底开不开放读取收藏列表？如果不开放，有哪些替代路径？哪条最现实？」——再去取证。最终结论「两家都按隐私法规锁死，个人/企业都申请不到」是先有问题才挖得出来的。

---

## 阶段 1 · 取证（一手优先于二手）

> 一条规则：**每个关键结论都要能指回一个来源。** 凭印象写的句子迟早会被打脸。

按对象类型选手段：

### 源码 / 仓库调研
- **通读关键路径源码**，不要只看 README。真正有价值的结论来自读码，例如「三个 adapter 都是一次性子进程、stdin 即关 → 没法跨家做真·运行中 steer」这种硬约束。
- 看 `package.json` / 入口 / 核心模块 / 数据流，画出真实架构。
- 区分「文档宣称的」和「代码实际做的」。

### 产品 / 设计调研
- 访问**真实产品 / 官网**，抓**官方截图**。
- 读官方博客、发布说明、设计规范原文。
- **网传二手说法要用一手来源校正**（例：网上传 Claude 是「蓝/紫」科技风，实测官方是暖米白 #F0EEE6 + 黏土橘 #D97757）。

### 技术 / 方案调研
- 官方文档、RFC、论文、benchmark 数据。
- 能动手跑就跑一个最小验证。
- 列清前提、依赖、版本。

### 可行性调研
- **先确认硬约束**（官方是否开放 / 是否合规 / 有没有配额），再谈方案。硬约束不成立的方案不要写进推荐。
- 给「行得通的路」排序，标注每条的代价。

### 现实事务 / 攻略调研
- 以**官方规则**为准（费用、流程、评分标准、时限），不要用过时的网络经验。
- 用 web 搜索拿最新口径，注明地区 / 时间（规则常按地区和年份变）。

### 导读 / 知识体系调研
- **回到原典**：尽量读一手文本（原著、权威译本、原始论文/讲稿），而不是只看「XX 思想精要」类二手转述。二手只用来辅助理解，结论以原典为准。
- **显式区分「原典说的」vs「通俗解读说的」**：很多流行说法是后人简化甚至误传，写之前先确认它在原典里站得住。
- **抓核心问题意识**：先搞清这套思想/著作要回应的是什么问题、在什么时代背景下提出——这是理解一切概念的钥匙。
- **拎概念、画脉络**：挑出最关键的 3–7 个概念，理清它们之间的推演关系（为后面的概念表和 Mermaid 脉络导图取证）。
- **收集常见误读**：搜「对 X 的常见误解 / 误读」，把「以为是 A，其实是 B」整理出来——这是导读最有价值的部分之一。
- 工具：`web_search` / `web_fetch` 找权威综述与原典出处；`view` 读用户给的原文文件。

### 取证工具
- 源码：`view` / `grep` / `glob`；大仓库可用 explore 子代理并行摸不同模块。
- 网络：`web_search`（要时效 / 权威口径）、`web_fetch`（读具体页面、抓官方原文）。
- 截图 / 媒体：按发布渠道的能力抓取并随文附上。

### ⚠️ 取证铁律：不存在的东西，绝不编（核心原则）

> **遇到陌生术语 / 缩写 / 概念，先核实它是否真实存在，再决定写不写。查不到实锤，就如实说「查不到这个东西」——绝不发明定义、绝不当成既有理论去展开。**

- 用户给的词可能是**口误 / typo / 记混**。不要默认它是一个成熟概念就替它造一套定义、画一张图、立一节内容。**先 `web_search` 核实**：它是不是一个真实存在、有出处的术语？
  - 查得到 → 用一手来源写，标出处。
  - 查不到 / 只查到无关结果 → **回头跟用户确认**「我没找到叫 X 的东西，你是不是指 Y / 是不是笔误？」，而不是硬编。
- **真实反例（务必内化）**：用户问「grain 这个范式呢」，其实是 typo、并无此概念。错误做法是把它当成「记忆粒度（memory granularity）」理论，造定义、单开一节、配一张「grain 谱系」图——**等于凭空发明了一个不存在的框架**。正确做法是先核实查无此术语，向用户确认是不是笔误。
- 区分两种东西：**「粒度 / granularity」是真实的工程考量**（可以写）；**把它包装成一个叫「grain」的命名理论**（与 GBrain / Karpathy 并列）就是编造。真实的概念照写，杜撰的命名和出处不要造。
- 同理：不要给结论造不存在的来源、不要把推测写成既成事实、不要给数字编一个看起来权威的出处。**宁可写「未找到 / 不确定」，也不要编一个像真的假信息。**

## 阶段 2 · 综合（从证据到结论）

下笔前先在脑子里（或草稿里）完成这三步，否则会写成「资料堆」：

1. **归纳成选项 / 路径**：把零散发现收敛成 2–4 个可比的选项或路径。
2. **做对比**：选项之间用统一维度对比（成本 / 复杂度 / 风险 / 适用场景…），这是对比表的来源。
3. **下判断**：给**明确推荐**——选哪个、为什么、放弃别的原因。调研文档的价值在结论，不在罗列。

同时准备好：**风险与缓解**、**下一步 / 落地建议**。

---

## 阶段 3 · 下笔

- 套 `references/doc-structure.md` 的标准骨架。
- **摘要（Executive Summary）最后写，但放最前面**——假设读者只看摘要也能拿到结论。
- 复杂内容用**对比表**、**Mermaid 导图画板**、**配图 / 截图**，不要用大段文字描述结构。
- 行文客观：证据 → 结论的链条要显式，不要让读者猜你凭什么这么说。

---

## 阶段 4 · 发布 / 归档 / 自检

- 按 `references/feishu-publishing.md` 发布（或按用户指定的 Markdown / 其它形式）。
- 发布后过一遍 `references/quality-checklist.md`。
- 给用户结论时**先说结论**（一句话核心发现），再给链接和结构概览。

---

## 节奏建议

- 简单调研（单一对象、问题清晰）：界定 → 取证 → 下笔，一气呵成。
- 复杂调研（多模块源码 / 多方案对比）：可先发一版骨架 + 关键发现，再补全；大仓库用 explore 子代理并行取证。
- **不要无限取证**。问题清单都回答了、关键结论都有来源了，就停下来下笔——别为了「再全一点」反复搜。
