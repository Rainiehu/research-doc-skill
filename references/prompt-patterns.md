# 调研 Prompt 指令模板 & 调试沉淀

这份文件把「怎么给 AI 下调研指令才能调得深」沉淀成可复用模板，并记录真实调试中踩过的坑。既给用户当 prompt 参考，也给执行调研的 agent 当自检。

---

## 一、好的调研指令长什么样

一条能调出深度的指令通常显式包含这几件事：

```
调研 <对象>，重点回答 <关键问题 1/2/3>，
基于 <取证手段：通读源码 / 实测产品 / 官方文档 / web 搜索>，
产出一篇 <形式：飞书文档 / Markdown>，
要有 <对比表 / 架构图 / 截图 / 推荐结论>。
```

### 真实指令样例（从实践中提炼）

- **源码型**：「调研一下这个 repo 到底是干嘛的，基于通读源码给我架构剖析和核心约束，写成飞书文档放知识库。」
  → 关键词「通读源码」「核心约束」逼出一手结论，而不是抄 README。
- **加功能型**：「调研给 lark-channel-bridge 加 Steer 功能的可行性，先读源码搞清现状语义，再给产品定义和方案。」
  → 先取证（读码）后下笔，结论里带「基于源码的架构剖析」。
- **产品型**：「调研 Claude 的设计语言，要官方一手的色值和截图，校正网上传的说法。」
  → 显式要求一手来源，避免抄二手。
- **可行性型**：「调研抖音/小红书能不能让 agent 读我的收藏，先确认官方开不开放，再给能走的路按推荐度排序。」
  → 先卡硬约束，再谈方案。
- **能力盘点型**：「调研 lark-cli 全部 skill 的能力，分类做总览表 + 一张分类导图，每个标名称/版本/作用。」
  → 指定了结构（总览表 + Mermaid 导图 + 字段），产出直接可用。
- **攻略型**：「调研悉尼 C 类驾照路考，按官方最新规则整理成攻略，覆盖资格/当天/流程/评分/挂科点。」
  → 指定时间线结构 + 官方口径。

### 让调研更深的「加料词」

把这些词加进指令，能显著提升深度：
- 「**通读源码 / 基于源码**」→ 逼一手结论
- 「**核心约束 / 硬限制**」→ 逼出真正决策相关的发现
- 「**官方一手 / 校正网传说法**」→ 防二手抄袭
- 「**先确认 X 是否成立，再谈方案**」→ 防在错误前提上展开
- 「**按推荐度排序 / 给明确推荐**」→ 逼结论而非罗列
- 「**对比表 / 架构图 / 每个配图**」→ 指定呈现形式

## 二、调试沉淀（真实踩坑 → 结论）

> 这些是把调研文档真正发出去、能用、归档好的过程中踩出来的。执行 agent 务必内化。

### 1. 标题坑（最高频）
- **`docs +create` 的 docxXML `<title>` 不生效** → 文档会建成 `Untitled`。
- **「整篇 overwrite」会把标题重置成 Untitled**。
- ✅ 结论：建完文档后**必须显式调 `update_title` 设标题**；不要靠 docxXML 的 `<title>`；尽量避免整篇 overwrite，必须 overwrite 后要重设标题并核验。

### 2. 身份坑
- 用 user 身份建的文档归属在用户名下，**没法改归属**，只能重建。
- user token 会过期（refresh token 也会过期），过期后建文档失败。
- ✅ 结论：默认 `--as bot`（机器人自己）建文档，归属稳定、不依赖用户登录态。

### 3. 归档坑
- `docs +create` 默认落在个人云盘，是**游离文档、不在知识库**。
- ✅ 结论：建完**立即 `wiki +move` 进知识库**（默认 AI Related），这是默认动作，不要问用户放哪。

### 4. 链接 unfurl 坑
- 流式 markdown 卡片里的链接**不会渲染成引用卡片**。
- ✅ 结论：要给文档链接时，**单独发一条 `msg_type:text` 的纯裸链接**让它 unfurl；正文里提一句「链接单独发你」。一条消息只放一个链接最稳。

### 5. 中文 shell 坑
- 命令行里直接出现中文（标题、JSON）会让 shell hook 挂起。
- ✅ 结论：要传中文的 title / JSON，**用文件落盘再 `@file` 引用**，命令行不出现中文。

### 6. 配图坑
- 纯文字描述结构读者记不住。
- ✅ 结论：结构上 Mermaid 画板，产品上官方截图，能力盘点每类配渲染示例图。生成后清理临时文件。

详细命令见 `feishu-publishing.md`。
