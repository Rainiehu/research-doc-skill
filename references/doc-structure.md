# 调研文档标准骨架

一套通用骨架 + 四种变体。**先固定骨架，再按对象类型微调分节。**

> 黄金法则：**摘要最后写、放最前面；每个关键结论指回一个来源；给明确推荐而不是罗列。**

---

## 通用骨架（默认用这个）

```
# [前缀] 标题：<对象> 调研 / <对象> 可行性分析

## 摘要 / Executive Summary        ← callout 块，最后写、放最前
- 一句话核心结论（读者只看这句也能决策）
- 2–4 条关键发现
- 推荐：选 X，因为 …

## 背景与目标 / Context & Goals
- 为什么做这个调研
- 目标 / 成功标准
- 范围 Scope（含已做的合理假设、不覆盖什么）
- 关键问题清单（这篇要回答的 3–5 个问题）

## 现状 / 核心分析                 ← 主体，按「关键问题」或「选项」分小节
- 每个选项/方面：描述 → 证据 → 评估
- 用对比表呈现差异
- 用 Mermaid 导图 / 架构图呈现结构
- 用截图 / 配图呈现产品与界面

## 对比 / 评估 Trade-offs
- 统一维度对比表
- 各选项强弱、适用场景

## 推荐 / 结论 Recommendation
- 选哪个、为什么（指回目标与证据）
- 放弃其它的原因
- 风险与缓解 Risks & Mitigations

## 下一步 / 落地建议 Next Steps     （如适用）
- 阶段 / 步骤、负责人、依赖

## 来源 / 参考 References
- 一手来源优先：源码路径、官方文档、官网链接、论文、RFC
- 二手来源标注清楚

## 附录 Appendix                   （可选）
- 原始数据、长表、代码片段、实测记录
```

---

## 变体一 · 产品 / 设计调研

侧重「它长什么样、为什么这么设计、我们能学什么」。

- **核心分析**拆成：定位与用户 / 信息架构 / 视觉语言（色彩·字体·组件）/ 关键交互 / 差异化。
- 必带**官方截图**与精确数值（色值、间距），不要用「大概是蓝色」这种话。
- 用一手产品/官网校正网传说法。
- 结尾给「对我们的启发 / 可借鉴点」。

## 变体二 · 源码 / 仓库调研

侧重「它实际怎么工作、有哪些硬约束、能不能扩展」。

- **核心分析**拆成：仓库定位（名字 vs 实际主体）/ 架构剖析（基于通读源码）/ 关键模块与数据流 / **核心约束**（来自读码的硬限制）/ 扩展点。
- 配一张 **Mermaid 架构图 / 流程图画板**。
- 明确区分「README 宣称的」与「代码实际做的」。
- 若调研是为「给它加某功能」，加一节：可行方案 + 改动面 + 风险。

## 变体三 · 可行性 / 方案调研

侧重「能不能做、有几条路、推荐哪条」。

- 先一节**硬约束 / 现状**（官方是否开放、是否合规、配额）——行不通的直接排除。
- **选项**逐条：描述 / 代价 / 风险 / 推荐度。
- 用对比表按推荐度排序。
- 结论给「按推荐度排序的真实可走的路」。

## 变体四 · 现实事务 / 攻略调研

侧重「按官方最新规则，该怎么准备、流程是什么、坑在哪」。

- 按**时间线 / 阶段**组织（考前资格 → 当天 → 流程 → 评分 → 常见挂科点）。
- 所有数字（费用、时长、时限）标注来源与适用地区/年份。
- 结尾给「速查清单 / checklist」。

## 呈现规范（让文档「好读」而不只是「完整」）

- **摘要用 callout 块**，第一眼就能拿到结论。
- **能用表就别用段落**：选项对比、特性对照、优劣权衡一律上表。
- **结构用图**：架构 / 流程 / 分类 / 时间线用 Mermaid 画板，比文字清楚十倍。
- **产品用图**：界面、视觉、成品用截图或本地渲染图随文附上（例：15 个图表场景，每个配一张实拍渲染图）。
- **分节自包含**：每节读者不用前翻也能看懂。
- 标题信息量要足：`<对象> 调研` 比「调研报告」强。

## 对比表模板

```
| 维度 \ 选项 | 选项 A | 选项 B | 选项 C |
|---|---|---|---|
| 成本 |  |  |  |
| 复杂度 |  |  |  |
| 风险 |  |  |  |
| 适用场景 |  |  |  |
| 推荐度 | ⭐⭐⭐ | ⭐⭐ | ⭐ |
```

对比要客观、要显式标出 trade-off，并在表下/推荐节点明「什么情况选哪个」。
